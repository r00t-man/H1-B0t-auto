# H1-B0t-auto

Telegram-бот для управления парком серверов [h1cloud](https://h1cloud.net) прямо из чата — список серверов, живые метрики, питание, продление аренды, обновление ядра Xray, перевыпуск REALITY-ключей, headless-клик «Создать новый конфиг» — плюс опциональная синхронизация CDN-гейтвея с панелью [Remnawave](https://remna.st) при его ротации на стороне провайдера.

Работает и без Remnawave (только управление h1cloud-серверами), и на голом сервере, и рядом с уже настроенной Remnawave-панелью.

> [!NOTE]
> Этот README написан с расчётом, что ты, возможно, впервые разворачиваешь Telegram-бота или впервые работаешь с h1cloud/Remnawave API. Если что-то кажется очевидным — пропускай, если нет — здесь должно быть объяснение.

## Что умеет

- **📋 Мои серверы** — список, живые метрики (CPU/RAM/аптайм), питание (старт/стоп/рестарт), баланс аккаунта, продление аренды.
- **🧬 Обновление ядра Xray** — программный эквивалент кнопки «Обновить ядро» в панели h1cloud.
- **🌐 Синхронизация CDN-гейтвея с Remnawave** *(опционально)* — при смене домена гейтвея на стороне провайдера бот сам подставляет новый адрес в Host-запись Remnawave, с TLS-проверкой перед применением.
- **🔑 Перевыпуск REALITY-ключей** *(опционально)* — скачивает новый ключ, точечно применяет только его (не весь ответ провайдера) в нужный inbound.
- **🌐 «Создать новый конфиг» одной кнопкой** *(опционально)* — у h1cloud нет API для этого действия, бот жмёт кнопку в панели сам через headless-браузер (Playwright).
- **🤖 Автоклик при восстановлении** *(опционально)* — следит за публичным каналом провайдера [t.me/h1cloud_status](https://t.me/h1cloud_status) и сам жмёт «Создать новый конфиг», когда провайдер объявляет о восстановлении доступа.
- **🗄 Legacy Pelican API** *(опционально)* — старый API прямого управления контейнером, для аккаунтов без доступа к новому Client API.

«Опционально» здесь значит: если ты не настроил конкретную интеграцию, соответствующая кнопка всё равно останется в меню бота — но при нажатии бот честно покажет, каких именно данных не хватает и в каком файле их дозаполнить (`.env` или `bindings.json`), вместо того чтобы просто исчезнуть или упасть с непонятной ошибкой.

> [!IMPORTANT]
> Мастер установки (`install.sh`) **всегда проходит по всем пунктам и спрашивает про каждую интеграцию**, с объяснением, зачем она нужна. «Опционально» относится к тому, что происходит, если ты нажмёшь `Enter` не отвечая на конкретный вопрос: этот пункт останется пустым, но кнопка в меню бота никуда не денется — заполнить недостающее можно позже (в мастере повторно или вручную в `.env`/`bindings.json`) и перезапустить бота. Подробный разбор — в разделе [«Первый запуск: чего ожидать»](#первый-запуск-чего-ожидать) ниже.

## Требования

- Ubuntu/Debian (apt) или RHEL/Fedora (dnf/yum) сервер, root-доступ (или `sudo`).
- Telegram-бот — создать через [@BotFather](https://t.me/BotFather) (см. ниже, если делаешь это впервые).
- Аккаунт h1cloud (для основного функционала).
- Remnawave-панель — только если нужна синхронизация CDN-гейтвея.

> [!TIP]
> Если ты никогда не создавал Telegram-бота: напиши [@BotFather](https://t.me/BotFather) команду `/newbot`, придумай имя и юзернейм (должен заканчиваться на `bot`, например `my_h1cloud_bot`). В ответ придёт **токен** вида `123456789:AAH...` — это и есть `TELEGRAM_BOT_TOKEN`. Никому его не показывай — с этим токеном можно полностью управлять твоим ботом.

## Установка

```bash
curl -L https://github.com/r00t-man/H1-B0t-auto/archive/refs/heads/main.tar.gz -o h1bot.tar.gz
mkdir H1-B0t-auto && tar -xzf h1bot.tar.gz -C H1-B0t-auto --strip-components=1
cd H1-B0t-auto
sudo bash install.sh
```

> [!TIP]
> Репозиторий публичный — ни SSH-ключ, ни `git` не нужны, скачивание работает на чистом сервере из коробки. Если `curl` недоступен, используй `wget`:
> ```bash
> wget -O h1bot.tar.gz https://github.com/r00t-man/H1-B0t-auto/archive/refs/heads/main.tar.gz
> ```
> Если предпочитаешь `git clone` — тоже подойдёт, по HTTPS ключ не требуется:
> ```bash
> git clone https://github.com/r00t-man/H1-B0t-auto.git
> ```

Скрипт сам:
1. Ставит `python3`, `python3-venv`, `python3-pip` через apt/dnf/yum (определяет пакетный менеджер автоматически).
2. Создаёт **venv** — изолированное окружение Python внутри папки проекта, чтобы зависимости бота не конфликтовали с другими программами на сервере — и ставит в него `requests`/`playwright`.
3. Запускает интерактивный мастер настройки (`h1bot/setup_wizard.py`) — по каждому пункту объясняет, зачем он нужен, `Enter` = пропустить этот конкретный пункт (не весь мастер).
4. Если в мастере указан логин панели h1cloud — ставит Chromium для Playwright (`playwright install --with-deps chromium`, скачивается ~300 МБ, один раз).
5. Генерирует и включает systemd-сервис `h1-b0t-auto` — это значит, что бот запустится автоматически при старте сервера и будет сам перезапускаться, если упадёт (`Restart=always`).

Повторный запуск `install.sh` безопасен — существующие `.env`/`bindings.json` не затираются молча, мастер предложит текущие значения по умолчанию (просто жми `Enter`, чтобы оставить как есть, и меняй только то, что нужно).

## Первый запуск: чего ожидать

Мастер (`h1bot/setup_wizard.py`) при первом запуске задаёт вопросы по порядку, **все семь разделов, без пропусков целиком**:

1. **Telegram (обязательно).** Токен бота + список Telegram id админов через запятую.
   > [!WARNING]
   > `ADMIN_IDS` — это не «имя пользователя», а числовой id. Узнать свой — написать [@userinfobot](https://t.me/userinfobot), он пришлёт в ответ число. Если оставить `ADMIN_IDS` пустым, бот будет отвечать «доступ запрещён» абсолютно всем, включая тебя.
2. **H1cloud Client API** — Bearer-ключ с `my.h1cloud.net/api-docs`. Если введёшь — мастер сразу же проверит ключ живым запросом и покажет имя аккаунта, чтобы ты убедился, что всё верно, не дожидаясь первого запуска бота.
3. **H1cloud panel — логин/пароль сайта.** Отдельно от API-ключа из пункта 2 — это обычные логин/пароль от личного кабинета `my.h1cloud.net`, которыми ты заходишь в браузере. Нужны только для одной кнопки, у которой физически нет API.
4. **Pelican API** (устаревший) — обычно можно смело пропустить, если ты не пользовался этой панелью раньше.
5. **Remnawave** — сначала да/нет, потом (если да) URL панели и токен.
6. **Автообнаружение привязок** — если и h1cloud, и Remnawave настроены, мастер сам скачает список твоих h1cloud-серверов и Remnawave-хостов и предложит выбрать пары номерами из списка (не нужно вручную копировать UUID).
7. **Автоклик** — просто предупреждение о том, что это такое; включается позже прямо в меню бота, по умолчанию всегда выключен.

> [!TIP]
> Не обязательно настраивать всё за один проход. Самый быстрый путь «просто посмотреть, что бот умеет» — заполнить только пункт 1 (Telegram) и пункт 2 (H1cloud Client API), а остальное пропустить нажатием `Enter`. Вернуться и дозаполнить можно в любой момент — см. раздел «Конфигурация» ниже.

## Удаление

Три равноценных способа снести бота — в консоли сервера, из самого бота или прочитать что удалится заранее в этом README:

- **Консоль (`uninstall.sh`).**
  ```bash
  sudo bash uninstall.sh                  # мягко: сервис + venv + кэш Chromium, .env/bindings.json остаются
  sudo bash uninstall.sh --wipe-secrets    # + стереть .env и bindings.json (реальные ключи/пароли)
  sudo bash uninstall.sh --remove-dir -y   # снести вообще всё, включая папку бота, без вопросов
  ```
  По умолчанию (без флагов) `.env`/`bindings.json` не трогаются — так можно быстро переустановить (`sudo bash install.sh`) без повторного ввода токенов и ключей.
- **Кнопка в боте.** «🗑 Удалить бота» в главном меню → подтверждение → бот останавливает и убирает свой systemd-сервис, чистит venv/кэш/секреты и сносит папку целиком (эквивалент `uninstall.sh --remove-dir -y`). Необратимо, как и остальные опасные кнопки — требует явного нажатия «✅ Подтвердить».
- **Полностью вручную**, если по какой-то причине не работают ни скрипт, ни бот:
  ```bash
  systemctl stop h1-b0t-auto && systemctl disable h1-b0t-auto
  rm -f /etc/systemd/system/h1-b0t-auto.service && systemctl daemon-reload
  rm -rf /путь/к/H1-B0t-auto
  ```

## Конфигурация

Два файла в корне проекта (ни один не коммитится в git — см. `.gitignore`):

- **`.env`** — секреты и простые настройки. Полный образец со всеми переменными и объяснением, зачем нужна каждая — `.env.example`. Можно редактировать вручную в любой момент, затем перезапустить сервис, чтобы изменения применились:
  ```bash
  systemctl restart h1-b0t-auto
  ```
- **`bindings.json`** — список привязок «какой h1cloud-сервер → какой Remnawave-хост» (для синхронизации CDN-гейтвея). Формат и пример — `bindings.example.json`. Может быть пустым списком `[]`, если синхронизация с Remnawave не нужна.

> [!TIP]
> Хочешь заново пройти мастер (например, чтобы добавить Remnawave, которого не было при первой установке) — не нужно переустанавливать всё с нуля, просто запусти его отдельно:
> ```bash
> cd /путь/к/H1-B0t-auto
> venv/bin/python3 h1bot/setup_wizard.py
> systemctl restart h1-b0t-auto
> ```

### Таблица переменных `.env`

| Переменная | Обязательна | Зачем |
|---|---|---|
| `TELEGRAM_BOT_TOKEN` | да | Токен бота от @BotFather. |
| `ADMIN_IDS` | да | Telegram id через запятую (числа, не username!) — кому разрешено пользоваться ботом. |
| `H1CLOUD_CLIENT_API_KEY` | нет | Bearer-ключ (`h1_...`) из `my.h1cloud.net/api-docs` — основной API: список серверов, метрики, питание, баланс, продление, обновление ядра, перевыпуск ключей. |
| `H1CLOUD_PANEL_LOGIN` / `H1CLOUD_PANEL_PASSWORD` | нет | Логин/пароль сайта `my.h1cloud.net` (НЕ API-ключ) — нужны только для headless-клика «Создать новый конфиг», у h1cloud нет API для этого действия. |
| `H1CLOUD_PELICAN_API_TOKEN` | нет | Токен старой панели `panel.h1cloud.net` — фолбэк для аккаунтов без доступа к Client API. |
| `REMNAWAVE_API_URL` / `REMNAWAVE_API_TOKEN` | нет | URL и токен твоей панели Remnawave — включает синхронизацию CDN-гейтвея/REALITY-ключей. |
| `GATEWAY_CHECK_INTERVAL` | нет (900) | Как часто (сек) проверять смену CDN-домена у привязанных серверов. |
| `CHANNEL_CHECK_INTERVAL` | нет (300) | Как часто (сек) проверять новые посты в `t.me/h1cloud_status`. |

> [!WARNING]
> `REMNAWAVE_API_URL` обычно должен включать суффикс `/api` в конце (например `https://panel.example.com/api`, а не просто `https://panel.example.com`) — так устроено большинство инсталляций Remnawave. Если после настройки бот пишет ошибки вида `HTTP 404` на действиях с Remnawave — в первую очередь проверь этот суффикс.

### Формат `bindings.json`

```json
[
  {
    "h1cloud_server_id": 1234,
    "label": "Например: EU-Gateway-1",
    "remnawave_host_uuid": "...",
    "remnawave_profile_uuid": "",
    "remnawave_node_uuid": "",
    "reality_inbound_tag": ""
  }
]
```

`remnawave_host_uuid` — обязателен для синхронизации CDN-домена. `remnawave_profile_uuid`/`remnawave_node_uuid`/`reality_inbound_tag` — опциональны, нужны только для кнопки применения перевыпущенных REALITY-ключей. Мастер установки может собрать этот файл интерактивно (подтягивает списки серверов/хостов через API и даёт выбрать пары), либо заполни вручную по этому образцу.

> [!NOTE]
> Не знаешь, где взять UUID хоста/профиля/ноды Remnawave вручную? Проще всего дать мастеру собрать `bindings.json` автоматически (пункт 6 при установке) — он сам запросит списки через API и покажет их человекочитаемо, без необходимости искать UUID вручную в базе данных или в другой панели.

## Архитектура

Один процесс, сырые HTTP-запросы к Telegram Bot API через `requests` (без `python-telegram-bot`/`aiogram`) — long-polling `getUpdates` + периодические фоновые проверки в том же цикле. Стейт (курсоры каналов, флаги автоклика, бэкапы перевыпущенных ключей) — плоские файлы в `state/`, без базы данных.

```text
H1-B0t-auto/
├── bot.py                 точка входа
├── install.sh             установщик: venv, зависимости, systemd, мастер настройки
├── uninstall.sh           деинсталлятор: сервис/venv/секреты/папка целиком
├── requirements.txt       requests, playwright
├── .env.example           образец конфига, с комментарием под каждую переменную
├── bindings.example.json  образец привязок h1cloud-сервер ↔ Remnawave-хост
│
├── systemd/
│   └── h1-b0t-auto.service.template
│
├── tests/
│   └── test_config.py       assert-тесты конфига и feature-флагов, без сети
│
└── h1bot/
    ├── config.py            .env + bindings.json → Config с feature-флагами
    ├── state.py             файловый key-value стейт (без БД)
    ├── telegram.py          тонкая обёртка над Bot API
    ├── http_utils.py        retry-обёртка над requests (h1cloud WAF отдаёт 403 на легитимные запросы)
    ├── notify.py            рассылка админам + cross-notify между ними
    │
    ├── h1cloud_client.py    H1cloud Client API v1 — серверы/метрики/питание/баланс/REALITY
    ├── h1cloud_pelican.py   H1cloud Pelican API — legacy, для аккаунтов без Client API
    ├── h1cloud_browser.py   Playwright: headless-клик «Создать новый конфиг» (нет API)
    ├── remnawave_client.py  минимальный клиент Remnawave (host/config-profile/node)
    ├── gateway_sync.py      синхронизация CDN-домена + REALITY-ключей по bindings
    ├── channel_watch.py     мониторинг t.me/h1cloud_status + автоклик
    │
    ├── keyboards.py         инлайн-клавиатуры — все кнопки видны всегда
    ├── handlers.py          роутинг callback'ов, паттерн `X_ask` → confirm → `X_go`
    ├── app.py               главный цикл: long-polling + фоновые проверки
    └── setup_wizard.py      интерактивный мастер настройки .env/bindings.json
```

## Безопасность

- `.env`/`bindings.json`/`state/` — в `.gitignore`, никогда не коммитятся.
- Бот отвечает только пользователям из `ADMIN_IDS`, всем остальным — «доступ запрещён».
- Необратимые действия (перевыпуск ключей, headless-клик «Создать новый конфиг», обновление ядра) требуют явного подтверждения кнопкой — бот сначала покажет предупреждение и попросит нажать «✅ Подтвердить», случайное нажатие одной кнопки ничего не сломает.
- `TELEGRAM_BOT_TOKEN`/`H1CLOUD_CLIENT_API_KEY`/пароли панели — в `.env` с правами `600` (файл читается только root).

> [!CAUTION]
> Кнопки «🔑 Перевыпустить REALITY-ключи» и «🌐 Создать новый конфиг (в панели)» **необратимы**: старый ключ/конфиг умирает сразу после нажатия, и все клиенты, у которых ещё не обновлена подписка, потеряют соединение до тех пор, пока не обновят её. Бот явно предупреждает об этом перед каждым таким действием — не игнорируй эти предупреждения «на автомате».
>
> Автоклик (пункт 7 мастера / кнопка-переключатель в меню сервера) делает это **без единого нажатия человеком**, когда в канале провайдера появляется пост о восстановлении. По умолчанию всегда выключен именно поэтому — включай осознанно, только если понимаешь последствия.

## Частые проблемы

- **Бот отвечает «⛔ Доступ запрещён» даже мне.** Проверь `ADMIN_IDS` в `.env` — там должен быть твой числовой Telegram id (не username), узнать его можно у [@userinfobot](https://t.me/userinfobot). После правки — `systemctl restart h1-b0t-auto`.
- **`playwright install --with-deps chromium` падает с ошибкой пакетов.** Флаг `--with-deps` сам ставит нужные системные библиотеки через apt — работает только на Debian/Ubuntu. На RHEL/Fedora ставь без `--with-deps` (`venv/bin/playwright install chromium`) и доустанавливай недостающие библиотеки по тексту ошибки вручную.
- **После настройки Remnawave бот пишет `HTTP 404`.** Почти всегда — не хватает `/api` в конце `REMNAWAVE_API_URL` (см. предупреждение в таблице переменных выше).
- **h1cloud иногда отдаёт HTTP 403 на ровном месте.** Это известное поведение WAF самой h1cloud на некоторые легитимные запросы — бот уже делает автоматический retry (`h1bot/http_utils.py`), обычно проходит со второй-третьей попытки без вмешательства.
- **Сервис не стартует после `install.sh`.** Смотри подробный лог:
  ```bash
  journalctl -u h1-b0t-auto -n 50 --no-pager
  ```
  Чаще всего причина — пустой `TELEGRAM_BOT_TOKEN` в `.env` (бот сразу завершается с понятным сообщением в лог).
